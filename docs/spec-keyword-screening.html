🔍 Keyword Screening — Spec

Sistema de screening de transações PIX que identifica termos restritos em descrições, detectando possíveis indícios de terrorismo, lavagem de dinheiro, fraude, crime financeiro e sanções.

Introdução

O Keyword Screening implementa uma regra de screening responsável por analisar a descrição de transações PIX e identificar a presença de palavras ou expressões restritas previamente cadastradas pela área de Compliance.

A solução utiliza um cache em memória dos termos restritos para garantir alta performance (< 10ms), suporta idempotência por transação e opera de forma resiliente mesmo quando o banco de dados estiver indisponível após a carga inicial.

Glossário

KeywordScreeningService — Serviço principal que orquestra a execução da regra
TextNormalizer — Normalização de texto (minúsculas, acentos, especiais, espaços)
KeywordMatcher — Identifica termos restritos na descrição normalizada
IdempotencyService — Garante processamento único por transação
RestrictedTermsCache — Mantém termos ativos em memória
RestrictedTermsScheduler — Recarrega termos periodicamente (5 min)
RuleExecution — Aggregate root: execução persistida da regra
Idempotency_Key — KEYWORD_SCREENING:{transactionId}

Requisitos

Requirement 1: Avaliação de Descrição PIX por Termos Restritos

Como analista de Compliance, quero que o sistema analise automaticamente a descrição de transações PIX em busca de termos restritos, para que possíveis indícios de atividades ilícitas sejam detectados em tempo real.

  1. WHEN requisição recebida com transactionId e description válidos → normalizar e verificar termos no cache
  2. WHEN pelo menos um termo encontrado → matched=true + lista de MatchResults
  3. WHEN nenhum termo encontrado → matched=false + matches vazia
  4. SHALL incluir ruleCode="KEYWORD_SCREENING" na resposta

Requirement 2: Normalização de Texto

Como desenvolvedor, quero que a descrição seja normalizada antes da análise, para que diferenças de formatação não impeçam a detecção.

  1. Converter para minúsculas
  2. Remover acentos → caracteres base
  3. Remover caracteres especiais (pontuação, símbolos)
  4. Compactar múltiplos espaços em um
  5. Pipeline sequencial: lowercase → acentos → especiais → espaços
  6. Propriedade de round-trip: resultado independe de variações de entrada

Requirement 3: Busca de Termos Restritos no Cache

Como analista de Compliance, quero que a busca seja feita exclusivamente em memória, para tempo de resposta < 10ms.

  1. Consultar exclusivamente o RestrictedTermsCache (sem DB durante avaliação)
  2. Comparar descrição normalizada com termos normalizados no cache
  3. Incluir termo e Category no resultado quando encontrado
  4. Tempo médio < 10ms para descrições até 140 caracteres
  5. Ignorar termos inativos (active=false)

Requirement 4: Gerenciamento do Cache

Como operador, quero que os termos sejam mantidos em memória e atualizados periodicamente.

  1. Carregar termos active=true no startup (fail-fast se banco indisponível)
  2. Manter em memória para consulta pelo KeywordMatcher
  3. Recarregar a cada 5 minutos via @Scheduled
  4. Continuar servindo termos anteriores se banco ficar indisponível
  5. Se reload falhar → manter cache + logar erro
  6. Armazenar termos já normalizados no cache

Requirement 5: Idempotência de Execução

Como analista, quero que a mesma transação não seja processada mais de uma vez.

  1. Verificar existência de RuleExecution com KEYWORD_SCREENING:{transactionId}
  2. Se existir → retornar resultado salvo sem re-executar
  3. Se não existir → executar, persistir e retornar
  4. Armazenar ScreeningResult completo no banco
  5. Race condition → permitir execução + limpeza via constraint UNIQUE
  6. UNIQUE(transaction_id, rule_code)

Requirement 6: API de Avaliação

Como sistema consumidor, quero uma API REST para submeter transações para screening síncrono.

  1. Endpoint: POST /v1/rules/keyword-screening/evaluate
  2. Requisição válida → HTTP 200 com ScreeningResult
  3. transactionId ausente/vazio/branco → HTTP 400
  4. description ausente/vazia → HTTP 400
  5. Aceitar description até 140 caracteres

Modelo de Dados

CREATE TABLE restricted_term (
    id         BIGSERIAL    PRIMARY KEY,
    term       VARCHAR(255) NOT NULL,
    category   VARCHAR(50)  NOT NULL,
    active     BOOLEAN      NOT NULL DEFAULT TRUE,
    created_at TIMESTAMP    NOT NULL,
    updated_at TIMESTAMP    NOT NULL
);
CREATE INDEX idx_restricted_term_active ON restricted_term(active);

CREATE TABLE rule_execution (
    id             BIGSERIAL    PRIMARY KEY,
    transaction_id VARCHAR(100) NOT NULL,
    rule_code      VARCHAR(20)  NOT NULL,
    result         JSONB        NOT NULL,
    created_at     TIMESTAMP    NOT NULL,
    CONSTRAINT uk_rule_execution UNIQUE(transaction_id, rule_code)
);
CREATE INDEX idx_rule_execution_lookup ON rule_execution(transaction_id, rule_code);
